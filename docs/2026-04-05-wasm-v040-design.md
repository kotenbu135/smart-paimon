# WASM v0.4.0 API Design Spec

## Overview

smart-paimonのアーキテクチャ原則「計算はすべてWASM」を達成するため、2つの新APIを追加する。
これにより、フロントエンドから約254行の計算ロジック（`character-buffs.ts` + `buildStatProfile`）を削除可能にする。

## API 1: `get_character_team_buffs`

### Purpose

キャラクター固有のチームバフ値をWASM側で計算して返す。
現在TypeScript側に実装されているBennett Q、Kazuha A4等のバフ計算ロジックをWASMに移行する。

### WASM Signature

```rust
#[wasm_bindgen]
pub fn get_character_team_buffs(
    json: &str,               // GOOD JSON全体
    character_id: &str,       // e.g. "bennett"
    constellation: u32,       // 凸数 (0-6)
    talent_levels: Vec<u32>,  // [normal, skill, burst] (1-15)
) -> Result<JsValue, JsError>
// Returns: Vec<ResolvedBuff> | JsError
```

`constellation`と`talent_levels`を明示的な引数にする理由: フロント側でシミュレーション（「C6にしたらどうなる？」）を可能にするため。GOOD JSONには実際のビルド情報（level, weapon等）が含まれる。

### Internal Function (good crate)

```rust
// crates/good/src/evaluate_talent_buffs.rs
pub fn evaluate_talent_buffs(
    build: &CharacterBuild,
    constellation: u8,
    talent_levels: &[u8; 3],
) -> Vec<ResolvedBuff>
```

純粋関数。WASMバインディングから呼ばれる。

**配置がgood crateである理由**: `build_stat_profile()`（good crate）を呼ぶ必要がある。data crateに置くとgood→dataの依存でdata→goodの循環依存が発生する。good crateはdata crateに依存済みなので、ここに置くのが自然。

**型について**: WASM境界では`Vec<u32>`/`u32`を受け取り、内部関数呼び出し前に`u8`/`[u8; 3]`に変換・バリデーションする。

### Processing Flow

1. `import_good(json)` → WASM引数の`character_id`に一致する`CharacterBuild`を取得
2. `find_talent_buffs(build.character.id)` → `Option<&[TalentBuffDef]>`を取得（Noneなら空Vec返却）
   - **Note**: GOOD JSONのキー（例: `"KujouSara"`）とdata crateのID（例: `"kujou_sara"`）は異なる名前空間。`import_good`が`CharacterBuild`を構築する際にdata crateの`CharacterData`を紐付けるため、`build.character.id`（snake_case）を使って`find_talent_buffs`を呼ぶ。WASM引数の`character_id`は`import_good`内でのビルド検索にのみ使用。
3. `build_stat_profile(build)` → `StatProfile`を取得（base_atk, base_def等のスケーリング元として使用）
4. 各`TalentBuffDef`を評価:
   a. **凸数チェック**: `def.min_constellation > constellation` → スキップ
   b. **突破段階チェック**: `AscensionPassive`ソースの場合、levelから突破段階を推定
      - Level >= 70 → ascension >= 4（A4パッシブ有効）
      - Level >= 40 → ascension >= 1（A1パッシブ有効）
      - `CharacterBuild`に`ascension`フィールドがないためlevelから推定
   c. **天賦倍率取得**: `def.source`から対応する天賦インデックスを決定
      - `TalentBuffSource::ElementalSkill` → `talent_levels[1]`
      - `TalentBuffSource::ElementalBurst` → `talent_levels[2]`
      - `TalentBuffSource::AscensionPassive` → 倍率テーブル不使用（`base_value`を使用）
      - `TalentBuffSource::Constellation(_)` → 倍率テーブル不使用（`base_value`を使用）
   d. **倍率値**: `talent_scaling`がSomeなら`scaling[talent_level - 1]`、Noneなら`base_value`
   e. **スケーリング計算**: `scales_on`がSomeなら対象ステータス値を乗算
      - `ScalingStat::Atk` → `profile.base_atk`（キャラ+武器の基礎ATK。`build_stat_profile`が算出済み）
      - `ScalingStat::Def` → `combine_stats(&profile).def`（total DEF）
      - `ScalingStat::Hp` → `combine_stats(&profile).hp`（total HP）
      - `ScalingStat::Em` → `combine_stats(&profile).elemental_mastery`（total EM）
      - `scales_on`がNone → 倍率値をそのまま使用
5. `ResolvedBuff`に変換: `source`, `stat`, `value`, `target`をセット
6. 対応していないキャラは空Vec `[]` を返す

### Scaling Stats Resolution

base_atk（Bennett, Sara）: `build_stat_profile(build).base_atk`を使用。
この値はキャラ基礎ATK + 武器基礎ATKの合算済み。`weapon_stat_index()`等の内部ロジックを重複させない。

total DEF/HP/EM（Yun Jin, Nahida等）: `combine_stats(&profile)`で最終値を算出。既存関数を再利用。

### Covered Characters (V1)

| Character | Source | Condition | Calculation | Stat |
|-----------|--------|-----------|-------------|------|
| Bennett | burst | always | base_atk × talent_scaling | AtkFlat |
| Bennett | c6 | C>=6 | fixed 0.15 | ElementalDmgBonus(Pyro) |
| Kazuha | a4 | level>=70 | EM × 0.0004 | DmgBonus |
| Nahida | a1 | level>=40 | min(max((EM-200)×0.25, 0), 250) | ElementalMastery |
| Mona | burst | always | talent_scaling | DmgBonus |
| Shenhe | skill | always | total_atk × talent_scaling | AtkFlat (既存TalentBuffDefの定義通り) |
| Shenhe | a4-press | level>=70 | fixed 0.15 | SkillDmgBonus |
| Shenhe | a4-press | level>=70 | fixed 0.15 | BurstDmgBonus |
| Kujou Sara | skill | always | base_atk × talent_scaling | AtkFlat |
| Kujou Sara | c6 | C>=6 | fixed 0.60 | CritDmg |
| Yun Jin | burst | always | total_def × talent_scaling | NormalAtkFlatDmg |

**Note**: Furinaはファンファーレスタック数がランタイム依存（HP変動イベント）のため、V1スコープから除外。
既存の`TalentBuffDef`データは保持し、将来ランタイム入力対応時にV2で追加する。

### Output Example

```js
// get_character_team_buffs(goodJson, "bennett", 6, [1, 1, 13])
[
  { source: "bennett:burst", stat: "AtkFlat", value: 756.0, target: "Team" },
  { source: "bennett:c6", stat: { "ElementalDmgBonus": "Pyro" }, value: 0.15, target: "Team" },
]

// get_character_team_buffs(goodJson, "diluc", 6, [10, 10, 10])
[]  // 対応外キャラは空配列
```

### Error Cases

- character_idがGOOD JSONに見つからない → `JsError`
- character_idがdata crateに存在しない → 空配列（未対応キャラ扱い）
- talent_levelsの長さが3でない → `JsError`
- talent_levelが範囲外(0 or >15) → `JsError`

## API 2: `build_member_stats`

### Purpose

GOOD JSONから`resolve_team_stats`の入力形式（`StatProfile`）を直接構築する。
フロントの`buildStatProfile()`（約100行）を置き換える。

### WASM Signature

```rust
#[wasm_bindgen]
pub fn build_member_stats(
    json: &str,           // GOOD JSON全体
    character_id: &str,   // e.g. "bennett"
) -> Result<JsValue, JsError>
// Returns: StatProfile | JsError
```

### Processing Flow

1. `import_good(json)` → character_idに一致する`CharacterBuild`を取得
2. `build_stat_profile(build)` → `StatProfile`を返す（既存関数）
3. `StatProfile`をJsValueとしてシリアライズして返却

### Output Example

```js
// build_member_stats(goodJson, "bennett")
{
  base_hp: 13103,
  base_atk: 544,     // character base + weapon base
  base_def: 784,
  hp_percent: 0.0,
  atk_percent: 0.466,
  def_percent: 0.0,
  hp_flat: 4780,
  atk_flat: 311,
  def_flat: 0,
  elemental_mastery: 0,
  crit_rate: 0.361,
  crit_dmg: 1.122,
  energy_recharge: 1.518,
  dmg_bonus: 0.0,
  pyro_dmg_bonus: 0.466,
  hydro_dmg_bonus: 0,
  // ... other element bonuses
}
```

### Error Cases

- character_idがGOOD JSONに見つからない → `JsError`

### Note

既存の`build_stats_from_good`は`Stats`（最終値）を返し、キャラ未発見時は`null`を返す。`build_member_stats`は`StatProfile`（分解値）を返し、キャラ未発見時は`JsError`を返す。この挙動差は意図的: `build_member_stats`はパイプラインの入力構築に使われるため、不在時は早期にエラーとすべき。

## Testing Strategy

### Unit Tests (data crate)

`evaluate_talent_buffs`を直接テスト:

- Bennett C0 burst lv13 → AtkFlat = base_atk × 1.19 (talent scaling)
- Bennett C6 burst lv13 → AtkFlat + ElementalDmgBonus(Pyro) 0.15
- Kazuha A4 with EM=800 → DmgBonus = 0.32
- Nahida A1 with EM=1000 → ElementalMastery = 200 (cap at 250, (1000-200)×0.25=200)
- Sara C0 → AtkFlat only; C6 → AtkFlat + CritDmg 0.60
- 未対応キャラ → empty Vec
- 凸数不足 → 該当バフをスキップ

### Integration Tests (wasm crate)

GOOD JSON文字列を入力として:
- `build_member_stats` → StatProfile値の検証
- `get_character_team_buffs` → ResolvedBuff配列の検証
- `build_member_stats`の出力が`resolve_team_stats`の入力として使えることの確認

## Frontend Migration (Reference)

### Before

```ts
const stats = buildStatProfile(build);        // TS calculation (~100 lines)
const charBuffs = getCharacterBuffs(build);   // TS calculation (~154 lines)
const member: TeamMember = {
  element, weapon_type, stats,
  buffs_provided: [...charBuffs, ...weaponBuffs, ...artifactBuffs],
  is_moonsign,
};
```

### After

```ts
const stats = build_member_stats(rawJson, characterId);
const charBuffs = get_character_team_buffs(
  rawJson, characterId, constellation, talentLevels
);
const member: TeamMember = {
  element, weapon_type, stats,
  buffs_provided: [...charBuffs, ...weaponBuffs, ...artifactBuffs],
  is_moonsign,
};
```
